# 국제화(i18n) 설계

선거 공약 아카이브에 외국인 사용자도 접근할 수 있도록 다국어 지원.

## 대상 언어
1. **한국어 (ko)** — 기본, 공약 원문 언어
2. **영어 (en)** — 1순위. 외국 언론·연구자 수요
3. **일본어 (ja)** — 재일 한국인 + 일본 교민
4. **중국어 간체 (zh)** — 재한 중국계 + 중국 언론

> 선관위 공식 자료 자체는 한국어로만 존재. 번역은 우리가 생성한 **보조 콘텐츠**임을 명확히 표기.

## 스코프 — 무엇을 번역하고 무엇을 번역하지 않는가

### ✅ 번역 대상
- **UI 정적 텍스트**: 헤더, 내비게이션, 버튼, 섹션 제목, 메타 문구, 에러 메시지
- **선거명·선거종류명**: "제8회 전국동시지방선거" → "8th Local Elections (2022)"
- **정당명**: "더불어민주당" → "Democratic Party of Korea"
- **공약분야 / 카테고리**: "주거" → "Housing"
- **공약 제목·요약** (AI 배치 번역, Phase 2)

### ❌ 번역하지 않음 (원본 보존 원칙)
- **공약 원문**: 선관위 제출 원문 그대로. 번역은 "번역 보기" 버튼으로 선택적 표시
- **후보자 이름**: 한글 + 로마자 병기 (Phase 2. 현재는 한글만)
- **주소, 선거구명**: 한글 그대로 + 영문 라벨 병기 (선택)

## 3단계 구현 로드맵

### Phase 1 (지금) — UI 인프라 + 정적 번역
- `next-intl` 도입, URL 패턴 `/{locale}/...`
- `dictionaries/{ko,en,ja,zh}.json` — UI 문자열 사전
- `LanguageSwitcher` 컴포넌트 (헤더)
- 루트 레이아웃의 `<html lang>` 동적 처리
- **번역 범위**: 랜딩, 선거 페이지, 후보자 페이지의 정적 UI만
- **공약 원문은 그대로 한국어**로 표시 (disclaimer 붙임)

### Phase 2 — AI 배치 번역
- 새 테이블 `pledge_item_translations` (lang, title, description, translated_at, translator)
- `ai/scripts/translate_pledges.py` — Claude API로 배치 번역
- 페이지에서 `locale != 'ko'`일 때 translations 우선 표시, 없으면 한국어 원문 + "번역 준비 중" 배지
- 번역 품질 보장: Claude Opus 4.6 사용, 중립 톤 유지

### Phase 3 — 후보자 고유명사
- `persons.name_romanized` 필드 추가 (McCune-Reischauer / Revised)
- 정당명 공식 영문 매핑 테이블
- 지역명 영문 표기 (예: 서울특별시 → Seoul Metropolitan City)

## 번역 톤 가이드라인
- **중립 유지**: 평가어·감정어 없음. "훌륭한 정책" X → "a policy that aims to..." O
- **정치 용어**: 한국 정치 맥락 설명 시 간단한 컨텍스트 추가 가능 ("지방선거" → "local elections, held every 4 years for regional/municipal officials")
- **고유명사**: 첫 등장 시 한자/한글 괄호 병기 권장
- **존댓말**: ko는 '-습니다', en은 declarative statements

## 디렉토리 구조 (Phase 1 완료 후)
```
app/
  [locale]/            ← 새로 추가, 모든 라우트를 감쌈
    page.tsx
    elections/[sg_id]/page.tsx
    candidates/[candidacy_id]/page.tsx
    ...
  layout.tsx          ← <html lang={locale}>
lib/
  i18n/
    config.ts          ← locales, defaultLocale, 타입
    dictionaries.ts    ← JSON 사전 로더
    dictionaries/
      ko.json
      en.json
      ja.json
      zh.json
middleware.ts          ← Accept-Language → /ko | /en 리다이렉트
```

## 브라우저 자동 번역과의 관계
- 페이지 언어를 `<html lang>`에 정확히 명시 → Chrome/Safari의 내장 번역이 최적 동작
- 공약 원문 블록은 `lang="ko"`로 명시 → 사용자가 원하면 브라우저가 해당 블록만 번역
- **Phase 1까지는 사용자 대부분이 브라우저 자동 번역으로 공약을 읽을 수 있게** 유도 (Phase 2 전까지의 현실적 솔루션)

## 주의사항
- **SEO**: 언어별 URL + `<link rel="alternate" hreflang="...">` 메타 필요
- **접근성**: `<html lang>` 변경 시 보조기기가 언어 전환 인지
- **RTL**: 현재 대상 언어 모두 LTR이라 무시 가능
- **날짜/숫자 포맷**: `Intl.DateTimeFormat`, `Intl.NumberFormat` 사용 — 한국식 "2026년 6월 3일" vs 영어식 "June 3, 2026"
